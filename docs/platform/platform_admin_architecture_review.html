A four-part technical review of the System Owner Console — what it can see and govern today, how the multi-tenant data layer is positioned for growth, and what stands between OpsTrax and its first serious third-party integration.

VerdictSOLID COREWORK TO SCALE

The platform admin control plane is architecturally sound and safe — tenant/platform isolation is real, mutations are audited, and the commercial control surface (lifecycle, billing, entitlements) is proven live. The gaps are ones of reach and readiness, not correctness: only 8 of ~30 modules are actually governable, the RBAC model out-promises what's wired, the database has no single source of truth, and there is no external integration surface yet. None block the pilot; all matter before scale.

Core finding — two disconnected vocabularies

The system has two parallel module-key systems, and only one is enforced. The runtime gate (ModuleKeyForPath in Program.cs) maps just 8 URL prefixes to keys and blocks disabled modules with a 403. A separate 17-key fleet.* billing catalog (RevenueSchemaService) exists purely for pricing display — its enforcement method CheckModuleAsync is called from exactly one place in the whole backend. The frontend's GATED_MODULES list mirrors the 8 enforced keys but matches neither the billing catalog nor the real product surface.

What this means for the CTO: a platform admin cannot enforce a lower-tier plan by disabling Vehicles, Drivers, Jobs, Fuel, Geofencing, Cold Chain, CRM leads, Notifications, Branches, Workforce, or AI Copilot — those modules have no gate to flip. Commercial packaging is only skin-deep until those gates exist.

Usage visibility is thinner than it looks

The PlatformRevenuePage is designed to show per-tenant usage, but of 13 seeded meters only 5 are ever written to. Vehicle, driver, shipment, user, asset and API-call counts are seeded but never incremented — they render as 0. Admin cannot see real operational counts anywhere; this needs either live COUNT(*) aggregates or RecordAsync wired into each module's create/update path.

02

Access control — a critical review

what admin should govern

You said "control everything." My counsel as CISO is control everything about the tenant as a business — and almost nothing inside the tenant's data. That line is the single most important design decision in a multi-tenant control plane, and OpsTrax already draws it correctly. The work is finishing the model, not widening it.

The governing principle

A platform admin who can silently read or edit any tenant's jobs, customers, and finances is not an admin — it's a breach waiting for an audit. The value of "everything" is governance (lifecycle, plan, limits, features, security posture, health), not operational reach (dispatching a truck, editing an invoice line). OpsTrax's existing standard enshrines this. Keep it.

Finding — the RBAC model out-promises the build

Nine platform roles are seeded (sales_admin, marketing_admin, support_admin, finance_admin, and more), but 7 permission keys they grant have zero enforcing endpoints: crm:*, marketing:*, proposals:*, support:view, impersonation:start, and health:manage. A Sales Admin or Support Admin can log in and hold permissions that gate nothing — not a security hole (the endpoints simply don't exist), but a credibility gap between the role model's ambition and the current surface. Either build the endpoints or trim the roles to what's real.

The one thing not to build carelessly: impersonation

Support access to live tenant data is the most-requested and most-dangerous capability. The platform_impersonation_sessions table and impersonation:start permission are reserved but deliberately unimplemented — the right call. When it is built it must be time-boxed, reason-required, read-only by default, and visible to the tenant it targets. Do not ship silent impersonation to close a support ticket faster; that single shortcut undoes the entire isolation model.

03

Database architecture for expansion

the foundation under it all

The tenancy model is sound and consistent — company_id everywhere, real composite indexes where migrations exist. But three structural issues will bite at scale, and one of them makes the current "flip a switch to enforce security" plan unsafe as-is.

FindingSeverityOne-line recommendation
No single source of truth for schema. ~80% of tables (≈177) exist only as runtime EnsureAsync() code with no migration file. A fresh RLS-enforced environment built "from migrations alone" would be missing most of the schema. Critical Generate migrations for all 32 remaining schema-services, or keep schema-init as an owner-role deploy job and treat migrations as its versioned record.
RLS enrollment is manual, not automatic. A new tenant table is invisible to Row-Level Security until someone re-runs the reconcile migration by hand. The maintainers already flagged "enroll-on-create" as unsolved. Critical Implement enroll-on-create (DDL event trigger or CI gate) before enabling RLS in production.
Telemetry / audit tables have no retention. Pure accumulation on a storage-billed Neon database; the retention doc is qualitative with no enforced thresholds or archival job. High Ship a scheduled partition-and-drop / delete-by-age job before telemetry volume compounds.
Historical isolation gaps recur by pattern. Past migrations fixed tables shipped with no company_id at all (Stage 24) and tables never RLS-enrolled (Stage 22). Nothing prevents the next module repeating either. High Add a CI lint that fails any PR introducing a tenant table without company_id + RLS enrollment.
Stale RLS exclusion list. The skip-list names platform_admin_users / platform_packages — tables that don't exist (real names: platform_admins, packages). Harmless today by coincidence; latent drift. Medium Replace hardcoded skip-list with a platform_* naming rule + a test asserting every entry resolves to a real table.
Untuned connection pooling. Relies on Npgsql defaults; the transaction-per-request RLS model consumes a connection for each in-flight request's full duration. Medium Set Max Pool Size explicitly and load-test the per-request transaction model before high-concurrency rollout.

The load-bearing insight

Today's production database connects as the owner role with RLS off, so EnsureAsync() runs on every boot and the runtime code is the working source of truth. That's fine now — but it means the RLS cutover is not a config flip. The moment you connect as the restricted role, schema-init stops running and the ~177 migration-less tables must already exist. Sequence the schema-source-of-truth fix before the RLS-enforcement fix, or the cutover breaks the database.

Adding a new module today — the copy-paste trap

There's no template or checklist, so a developer cloning an existing schema-service would likely forget: the migration file (table missing under RLS), RLS re-enrollment (cross-tenant leak once RLS is on), the entitlement gate (feature given away free), and audit wiring. A one-page "new tenant module" checklist plus a scaffolding template would close the most expensive recurring gap on the team.

Integration readiness

open to everything, honestly

You asked to be open to any integration. The right posture is exactly that — with clear eyes about what exists. One integration surface is genuinely production-shaped; the rest range from strong plumbing missing its last mile, to documented-only, to a mock catalog that must not be mistaken for real.

SurfaceStateReality
IoT / hardware (GPS, ELD)Production-shapedReal HMAC device auth, provisioning, rotation, revocation, freshness + nonce checks. A vendor could integrate today. Strongest surface.
Outbound webhooksPlumbing onlyDurable transactional outbox + dispatcher is built — but zero external HTTP delivery, no subscription model, no signing secrets. The hard half is done; the outward half is absent.

Email / SMSIn-app onlyNo Twilio/SendGrid/Postmark. Notifications write to a table; nothing sends outbound.

Cross-cutting blockers for any external consumer

Three things gate every integration category regardless of type: no API versioning (all routes are bare /api/* — one breaking change hits the SPA, mobile, and partners simultaneously), no partner API-key or OAuth client-credentials model (the only key mechanism is IoT-device-scoped), and IP-based rather than key-based rate limiting (a NAT'd device fleet gets throttled like an attacker). Fix these before onboarding any external system.

Recommended sequence

CTO priorities

Ordered by dependency and risk, not ambition. The P0 items are prerequisites for decisions already on the roadmap (RLS enforcement, commercial packaging); the rest compound value once the foundation holds.

  • P0 · FoundationResolve the schema source-of-truth before enforcing RLS. Generate migrations for the 32 remaining schema-services (or make schema-init a deploy job). Without this, the RLS cutover breaks the database.
  • P0 · SecurityShip enroll-on-create for RLS so no future table can silently skip tenant isolation, then enable RLS in production.
  • P1 · ProductExtend the entitlement gate to the ungated core modules (Vehicles, Drivers, Jobs, Fuel, CRM leads, AI) so commercial packaging is actually enforceable — and wire the dormant usage meters to real counts.
  • P1 · AccessReconcile RBAC with reality: build the missing endpoints for the 7 dangling permissions or trim the roles. Design the audited impersonation flow before support demands it.
  • P2 · GrowthLay integration groundwork: API versioning (/api/v1), a partner API-key/OAuth model, and the outbound webhook delivery handler — the three unlocks that make every future integration cheaper.
  • P2 · ScaleRetention & pooling: a telemetry archival job and explicit pool sizing with a load test, before volume forces the issue.
