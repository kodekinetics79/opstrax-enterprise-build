# OpsTrax — SME Product Audit & Superiority Backlog

_Generated from an autonomous 15-lane domain-expert audit (9 module + 6 persona lanes). Each expert audited the **real** code (no seed/fake assumptions; reliance on seed data is itself flagged as a gap). 9 lanes were additionally code-verified by a skeptical second pass that dropped already-implemented items; 6 persona lanes are audit-only (verification was cut short by a session limit) and are marked accordingly._

**Totals:** 168 gaps — P0: 7, P1: 71, P2: 70, P3: 20

## Cross-cutting themes (CTO synthesis)
1. **Flagship features are display shells over seed data.** Predictive Analytics ('ML Models Active'), AI Copilot, Control Tower, HOS/ELD, AI dashcam, geofencing, and predictive maintenance render seeded/static content with no engine behind them. Several self-label 'Live' — a credibility and (for 'ML Models Active') a truth-in-advertising risk.
2. **Real-time is actually near-time.** Dispatch board is ~30s stale; safety/crash/SOS alerts are batched on a 5-min tick with no push. No predictive ETA is derived from live telemetry anywhere.
3. **Finance is a set of disconnected sub-ledgers.** No GL/double-entry/ERP posting, no AR credit notes, multi-currency stubbed, US sales tax unsupported, e-invoicing (ZATCA Ph2) absent, rev-rec point-in-time only, and maker-checker permits self-approval (no SoD).
4. **Telematics breadth is one protocol deep.** Only GT06; the device gateway is built but not deployed (raw trackers can't reach prod); geofencing is display-only; no ELD/HOS engine.
5. **Security has a login-bricking P0.** MFA-at-login is enforced as a lockout with no enroll/challenge/verify path; impersonation is schema-only; session timeouts stored but not enforced.
6. **Dead links / dead endpoints in real journeys.** CRM Leads/Opportunities/Campaigns 404; customer-ETA tracking link is dead; the dispatcher's core 'assign a load' action has no UI.

### Customer (persona journey) — shipper/consignee visibility, ETA, exception notifications, proof of delivery, self-service portal  ⚠️ audit-only
_The customer persona is one of the more genuinely built-out journeys in the repo, but it is architecturally fragmented and has one hard end-to-end break plus a hollow "notifications" promise. There are THREE parallel, overlapping public tracking backends: (a) Fleet-TMS `/api/public/shipments/track/{token}` over `fleet_tms_tracking_links` (128-bit Guid tokens), (b) `/api/customer-visibility/tracking/{token}` over `customer_visibility` (256-bit hex tokens, expiring/revocable), and (c) `/api/customer-eta/track/{trackingCode}` over `customer_eta_links` (256-bit secure_token). The security engineering on the public paths is genuinely strong: tokens are unguessable, expiring and revocable; the anon bypass in Program.cs is GET-only and scoped; POD assets are never returned as raw URLs but proxied through a token-revalidated, SSRF-hardened, host-allowlisted streaming proxy (FleetTmsEndpoints.cs PublicTrackPodAsset); customer-safe projections strip driver/HOS/internal fields (SafeCustomerShipmentView, BuildCustomerTimeline). The authenticated portal (CustomerPortalEndpoints/CustomerPortalService) is correctly double-scoped by company_id AND customer_id, and frontend services are pure-live (react-query with real error/empty states, no withFallback seed). However: (1) the flagship 'Send ETA update -> hand customer a secure link' flow produces a DEAD LINK because SendEta and CustomerEtaPage build `/track/{token}` which resolves against the Fleet-TMS table, while the token lives in customer_eta_links whose only working page is `/eta/{token}`; (2) 'exception notifications' are a database illusion — SendEta writes eta_updates with status='Sent' and channel='Email/SMS' but no email/SMS is ever dispatched, NotificationService only creates in_app rows, and the real Twilio connector is wired only to a manual 'send-test' action; (3) ETA is not predictive — ComputeEtaAsync returns the static planned_delivery_at as the ETA, never computing from live GPS + remaining route + traffic; (4) no customer-facing live map/vehicle pin exists anywhere. So: strong security bones and a real portal, undermined by fragmentation, a broken primary link, and notification/ETA capabilities that are simulated rather than delivered._

**Real-scenario coverage:** Partially real, with real integration-test backing on the security-critical slices but broken/simulated on two headline scenarios. REAL and tested: cross-tenant + cross-customer isolation of the portal is verified against real Postgres (CustomerPortalPostgresTests: PortalInvoices_AreScopedToTheCustomer_NotJustTheCompany, PortalJobDetail_StripsInternalFields_AtTheQueryLevel, PortalProofs_AreCustomerSafe_AndScopedToTheCustomer); the customer-portal-user RBAC boundary is unit-tested (CustomerPortalAuthBoundaryTests — portal principal rejected from every internal permission even when it overlaps shipments:view); the ETA staleness regression (nonexistent recorded_at column that 500'd every tracker) is pinned (CustomerVisibilityEtaRegressionTests); CustomerVisibilityTests has ~25 facts. The public token track/events/pod paths and the POD asset proxy are real code with no seed fallback (customerVisibilityApi.ts, fleetTmsApi publicTrackingApi, PublicShipmentTrackingPage.tsx render live or show a clean 'link unavailable' error). NOT real / broken: the customer-eta 'send update -> track via link' scenario is broken end-to-end — SendEta (EndpointMappings.cs:4932) returns trackingUrl `/track/{token}` and CustomerEtaPage.tsx:127 copies `${origin}/track/${token}` to clipboard, but `/track/:token` (App.tsx:178) renders PublicShipmentTrackingPage which queries fleet_tms_tracking_links, NOT customer_eta_links; the only page that resolves that token is `/eta/:trackingCode` (App.tsx:177). Exception/ETA notification delivery is simulated: no code path invokes Twilio/SMTP for customer updates; SendEta just flips status='Sent'. - **[P1/S] Customer-ETA secure tracking link is a dead link (wrong route prefix / wrong backend table)** (correctness)
  - The primary 'ETA & Communications' flow lets a dispatcher click 'Send ETA update', which mints a 256-bit secure_token in customer_eta_links and returns trackingUrl = `/track/{token}`; CustomerEtaPage independently builds `${origin}/track/${token}` and copies it to the clipboard as 'secure tracking link'. ### Dispatcher (persona journey)  ⚠️ audit-only
_The dispatcher journey is genuinely half-built and half-hollow. The backend assignment lifecycle is real, not seed: POST /api/dispatch/assignments runs a true multi-factor eligibility gate (vehicle OOS hard-block, critical DVIR defects, blocking work orders, overdue PM warnings, driver status, tenant-configurable safety-score threshold, HOS snapshot, unresolved critical safety events, clamped match score), then create/accept/status/exception/cancel/proof all route through ONE shared write path (ApplyAssignmentTransitionAsync, EndpointMappings.cs:15354) that atomically mirrors assignment_status, the display status, the linked job, trip state and an idempotent job.delivered outbox event — a real fix for the classic two-write-path drift. The board endpoint (EndpointMappings.cs:3890) buckets live assignments plus unassigned jobs into a canonical Kanban and normalizes legacy title-case statuses. DispatchCommandPage.tsx is a real, permission-gated UI wired to these endpoints for monitoring, advancing, flagging exceptions, recording proof and cancelling. That is the strong half. The hollow half: the single most important dispatcher action — pairing an unassigned load with a driver+vehicle — has NO frontend at all (dispatchApi.createAssignment is defined but never called anywhere in the app); the board's Unassigned column and the Eligibility tab can look but cannot assign. The 'smart assign'/recommendations layer is not an optimizer (seed-populated table + placeholder match reasons + a trivial additive score over a client-supplied driver/vehicle). There is no proximity/ETA-to-pickup awareness anywhere. And the dispatch test suite creates a false sense of safety: the eligibility, tenant-isolation and lifecycle 'tests' assert on inline reimplementations and tautological constants rather than calling the real handlers. Net: a dispatcher can watch and progress a shift that someone else somehow started, but cannot actually start it, cannot be told who is the best/closest driver, and gets board updates only every 30 seconds._

**Real-scenario coverage:** Partial and asymmetric. The MONITOR/ADVANCE/EXCEPTION/PROOF/DELIVER half of a real shift is backed by real code paths (endpoint -> shared state machine -> SQL mirroring job+trip+outbox -> real UI), not withFallback/seed. The ASSIGN half is not usable end-to-end: createAssignment has no UI, so 'see board -> assign load' cannot be executed in the product (grep confirms createAssignment/acceptAssignment unused outside driver app). The INTELLIGENCE layer relies on seed/placeholder: dispatch_recommendations is only populated by Batch2SchemaService.cs:275 and seed 002_seed.sql:366 (no engine ever inserts), DispatchAutoSuggest returns hardcoded 'HOS risk placeholder'/'Proximity placeholder' strings (EndpointMappings.cs:5059), and Stage9 RecommendSmartAssignment just stores a client-provided pairing with a fixed additive score (Stage9OperationalFoundationService.cs:1698). The one Postgres e2e that touches dispatch inserts a pre-'delivered' assignment via raw SQL (OrderToCashE2EPostgresTests.cs:113), bypassing every lifecycle endpoint. So the create->accept->en_route->exception->deliver flow through the HTTP surface has zero real integration coverage.
