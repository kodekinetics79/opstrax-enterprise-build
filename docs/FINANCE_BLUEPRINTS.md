## GL auto-post on invoice issue  (`gl-autopost-on-issue`)  — effort M

**Summary:** Auto-post every issued AR invoice into the double-entry GL at issue time by adding a second IOutboxMessageHandler on the existing durable `invoice.issued` outbox event that calls GeneralLedgerService.PostInvoiceAsync. Because the outbox registry today allows only ONE handler per event type and `invoice.issued` is already claimed by InvoiceIssuedRecognitionHandler, the enabling change is to make the handler registry + dispatcher fan-out (invoke ALL handlers for an event, all-or-retry). Reuse of the already-idempotent PostInvoiceAsync means the ledger self-populates in near real-time (subledger->GL, NetSuite/SAP style) with no schema or endpoint changes.

**Current state:** Invoices are issued in RevenueReadinessService.IssueInvoiceAsync; after the DB transaction commits it publishes the durable event via events.Publish("invoice.issued", aggregateType "issued_invoice", aggregateId = issued.Id [the invoice UUID as string]) at RevenueReadinessService.cs:907-923. That event is drained by OutboxDispatcherBackgroundService (OutboxDispatcherBackgroundService.cs:31-40, runs under db.RunInSystemScopeAsync platform-admin bypass to drain all tenants) -> PostgresOutboxDispatcher.DispatchOutboxOnceAsync (FoundationDispatcherServices.cs:116-153). The dispatcher resolves EXACTLY ONE handler via outboxHandlers.Resolve(eventType) (line 132) and marks the whole message processed/failed on that single handler. The registry OutboxMessageHandlerRegistry builds `handlers.ToDictionary(h => h.EventType, OrdinalIgnoreCase)` (FoundationDispatcherServices.cs:48-49) — one handler per event type; a duplicate EventType throws ArgumentException at singleton construction (boot crash). `invoice.issued` is already taken by InvoiceIssuedRecognitionHandler (InvoiceIssuedRecognitionHandler.cs:9-19, registered Program.cs:245) which drives the rev-rec subledger. GeneralLedgerService already exists and is DI-registered (Program.cs:237); PostInvoiceAsync(companyId, invoiceId) reads issued_invoices (subtotal, tax_total, total, issued_at, invoice_number) and posts Dr 1100 total / Cr 4000 subtotal / Cr 2200 tax, idempotent via UNIQUE(company_id, source_type, source_ref) on journal_entries (GeneralLedgerService.cs:92-109, schema GeneralLedgerSchemaService.cs:34). NO GL handler on invoice.issued exists today, so issued invoices never reach the ledger automatically — the trial balance stays empty unless PostInvoiceAsync is called manually. GL tables carry company_id BIGINT and are auto-enrolled in RLS by RlsReconciliationSchemaService at boot-final (Program.cs:484). The only caller of registry.Resolve is the dispatcher (FoundationDispatcherServices.cs:132); RegisteredEventTypes has no external consumer — so the fan-out change has a contained blast radius.

**Algorithm:**

FAN-OUT REGISTRY (enabling change):
1) IOutboxMessageHandlerRegistry (FoundationContracts.cs:383-387): add `IReadOnlyList<IOutboxMessageHandler> ResolveAll(string eventType)`; keep Resolve for back-compat.
2) OutboxMessageHandlerRegistry (FoundationDispatcherServices.cs:46-55): replace the ToDictionary with a group-by: `_handlers = handlers.GroupBy(h => h.EventType, StringComparer.OrdinalIgnoreCase).ToDictionary(g => g.Key, g => (IReadOnlyList<IOutboxMessageHandler>)g.ToList(), StringComparer.OrdinalIgnoreCase)`. This removes the duplicate-key crash. ResolveAll returns the list or Array.Empty; Resolve returns list.FirstOrDefault; RegisteredEventTypes = _handlers.Keys.
3) Dispatcher loop (FoundationDispatcherServices.cs:132-149): `var handlers = outboxHandlers.ResolveAll(message.EventType); if (handlers.Count == 0) { HandleOutboxFailure("No handler registered for {EventType}"); continue; }`. Then invoke each in registration order inside one try/catch: `foreach (var h in handlers) { try { await h.HandleAsync(message, ct); } catch (Exception ex) { throw new InvalidOperationException($"{h.GetType().Name}: {ex.Message}", ex); } }`. Only after ALL succeed: MarkOutboxProcessed + eventLogs success. If any throws -> catch calls HandleOutboxFailureAsync(message, ex) (retry_pending w/ exponential backoff, or dead_letter at MaxRetryCount) and continue. Message is marked processed at-most-once; succeeded handlers re-run on retry (safe because every handler is idempotent). The wrapping preserves the failing handler's type name in last_error/dead_letter_reason for audit.

**Idempotency / fail-closed:** Idempotent at three layers: (1) outbox is at-least-once but PostInvoiceAsync early-returns the existing entry id when a row already exists for UNIQUE(company_id,'invoice',invoiceId) (GeneralLedgerService.cs:56-60) — duplicate invoice.issued delivery = exactly one journal entry; (2) the fan-out dispatcher marks a message processed at-most-once and only after ALL handlers succeed, so a partial failure re-runs the already-succeeded GL/rev-rec handlers, both safe; (3) EnsureChartAsync uses ON CONFLICT DO NOTHING. Fail-closed: PostEntryAsync throws unless debits==credits, so an invoice whose total != subtotal+tax_total is NEVER posted (it retries then dead-letters, surfacing the bad data instead of corrupting the ledger); a missing issued_invoices row throws (GeneralLedgerService.cs:97) -> retry/dead-letter; a malformed aggregateId throws (visible) rather than silently no-op. No config/seed path can cause a silent wrong post. The upstream issuance transaction already committed and is never rolled back by a handler failure (derive-beside): the outbox retries with exponential backoff and dead-letters after MaxRetryCount (default 3), recording last_error/dead_letter_reason prefixed with the failing handler's type name for audit.

**Tenant-scoping / security:** companyId is derived solely from message.TenantId (set by the publisher to companyId.ToString(), RevenueReadinessService.cs:908) — a handler can never post to another tenant's books. Every SQL in PostInvoiceAsync/PostEntryAsync/EnsureChartAsync filters by company_id (GeneralLedgerService.cs:58-59,95, etc.), and the read of issued_invoices is `WHERE id=@id::uuid AND company_id=@c` so a UUID from tenant A cannot pull tenant B's invoice. chart_of_accounts/journal_entries/journal_lines all carry company_id BIGINT and are auto-enrolled in RLS by RlsReconciliationSchemaService (Program.cs:484); the dispatcher intentionally runs under db.RunInSystemScopeAsync (platform-admin bypass, OutboxDispatcherBackgroundService.cs:31) because it drains all tenants, so isolation rests on the explicit per-query company_id predicates (verified present) rather than RLS at drain time. No new PII, no new auth surface. If the optional backfill endpoint is added it must be company-scoped and gated maker-checker (high-risk finance action) consistent with finance.invoice.issue.

**Tests to write:**
- Handler posts on issue: publish invoice.issued for a seeded issued_invoice (subtotal 200, tax 30, total 230), run DispatchOutboxOnceAsync with a registry containing InvoiceIssuedGeneralLedgerHandler -> assert Dr 1100=230, Cr 4000=200, Cr 2200=30, trial balance nets zero, exactly one journal_entries row (source_type='invoice', source_ref=id), message status='processed'
- Idempotent double-delivery: dispatch the same message twice (or publish twice) -> still exactly one journal entry, entry ids equal
- Zero-tax invoice: total==subtotal, tax_total=0 -> only two lines (no 2200), entry still balances and posts
- Fan-out both handlers: register BOTH InvoiceIssuedRecognitionHandler and InvoiceIssuedGeneralLedgerHandler for invoice.issued, dispatch once -> BOTH the rev-rec subledger row AND the journal entry are created, event_processing_logs has one success, outbox status='processed'
- Registry no longer crashes on duplicate EventType: constructing OutboxMessageHandlerRegistry with two invoice.issued handlers does not throw
- Partial-failure retry semantics: rev-rec handler succeeds but GL handler throws (invoice row absent) -> message status='retry_pending', retry_count=1, NOT processed; then insert the invoice row and re-dispatch -> GL posts, rev-rec did not duplicate, exactly one journal entry, status='processed'
- Fail-closed imbalance: seed an issued_invoice with total != subtotal+tax_total -> dispatch -> handler throws, NO journal entry, message retry_pending/dead_letter, last_error mentions 'does not balance' and the handler type name
- Tenant scoping: two companies each issue an invoice; assert each company's journal_lines contain only its own entry and PostInvoiceAsync for company A cannot read company B's invoice (cross-tenant id -> throws not-found)
- Malformed aggregateId: publish invoice.issued with a non-GUID aggregateId -> handler throws, message goes to failure (not silently processed)

**Edge cases:**
- Duplicate invoice.issued delivery (outbox at-least-once) -> idempotent single entry via UNIQUE(company_id,source_type,source_ref)
- Partial fan-out failure re-runs succeeded handlers on retry — safe only because rev-rec AND GL are both idempotent; if a future non-idempotent handler is added to invoice.issued this all-or-retry model would double-apply it (document the invariant)
- issued_invoices.total != subtotal+tax_total (rounding/data drift) -> PostEntryAsync throws, dead-letters instead of posting an imbalanced entry; reconcile issuance totals before enabling
- Tax=0 -> no 2200 line (handled by PostInvoiceAsync)
- Multi-currency: GL/journal_lines have NO currency dimension; issued_invoices.currency is ignored, so mixing currencies within one company's trial balance is arithmetically wrong — document as a known limitation / follow-up (add currency to journal_lines + per-currency TB)
- Credit notes / negative or reversing invoices: PostInvoiceAsync books Dr AR = total which may be negative; no dedicated reversal handling — out of scope, note for AP/credit-note workstream
- Production GL schema gate: GeneralLedgerSchema:Enabled defaults false in prod (Program.cs:457) -> tables missing -> every post dead-letters; must set flag or ship owner migration BEFORE enabling
- Outbox dispatcher disabled in prod: OutboxDispatcher:Enabled/AllowProduction must be on or events never drain and the ledger silently stays empty (a fallback/no-op-style defect) — verify config
- Backfill of pre-existing issued invoices is required for a complete ledger; go-forward handler only covers newly issued invoices
- Handler ordering: rev-rec and GL are independent (order-agnostic), but they now share one outbox message, so a persistently failing rev-rec handler would also block the GL success mark (and vice-versa) until it dead-letters — acceptable given idempotency, but monitor dead_letter_reason by handler name

PERIOD MODEL. HARD LOCK (no back-posting) = DB trigger, the real guardrail so ANY writer (PostInvoiceAsync, future AP/tax/manual posters) is blocked, not just the service:
CREATE OR REPLACE FUNCTION gl_enforce_period_lock() RETURNS trigger AS $$
BEGIN
  IF EXISTS (SELECT 1 FROM gl_periods p WHERE p.company_id=NEW.company_id AND p.status='closed' AND NEW.entry_date BETWEEN p.period_start AND p.period_end) THEN
    RAISE EXCEPTION 'gl_period_closed: entry_date % is in a locked period for company %', NEW.entry_date, NEW.company_id USING ERRCODE='P0001';
  END IF; RETURN NEW; END; $$ LANGUAGE plpgsql;
CREATE TRIGGER trg_gl_period_lock BEFORE INSERT ON journal_entries FOR EACH ROW EXECUTE FUNCTION gl_enforce_period_lock();
(The trigger's SELECT is RLS-safe: it runs in the app session with app.current_tenant_id set; the explicit company_id=NEW.company_id predicate keeps it correct even under platform-admin bypass.) App-level mirror in PostEntryAsync for a friendly error before the DB round-trip: after the balance check, SELECT 1 FROM gl_periods WHERE company_id=@c AND status='closed' AND @entryDate BETWEEN period_start AND period_end; if found throw PeriodClosedException -> callers surface reason 'period_closed'.

## Telemetry-driven predictive ETA  (`predictive-eta`)  — effort L

**Summary:** Replace the planned-time ETA heuristic with a telemetry-driven predictive engine: ETA = now + remaining_road_distance / effective_speed + dwell_buffer, where remaining distance comes from the live vehicle position to the drop-off (Google Directions when a key is configured, else Haversine × per-tenant circuity factor), effective_speed blends smoothed live speed with a historical per-lane min/mile median, and dwell_buffer covers remaining stops. A new PredictiveEtaService + eta_predictions/eta_lane_stats/eta_prediction_accuracy tables and a 3-minute background recompute replace ComputeEtaAsync's plannedDelivery return, with the existing heuristic kept as the fail-closed fallback (no live fix / no coords / no history => planned time at confidence=low, never fabricated). An accuracy loop records predicted-vs-actual on delivery to feed the "ETA Accuracy" SLA, matching project44/FourKites behaviour.

**Current state:** ComputeEtaAsync (backend-dotnet/Controllers/EndpointMappings.cs:16080-16201) is the whole "engine". It NEVER predicts: line 16155 sets `etaAt = plannedDelivery` verbatim, and the only telemetry it touches is a staleness boolean (16107-16134) that queries latest_vehicle_positions.received_at solely to downgrade confidence — lat/lng/speed are never read. Risk is pure date arithmetic vs plannedDelivery (16153-16184). QuickEtaRisk (16205-16218) is the list-view variant: status + planned date only. **Algorithm:**

PredictiveEtaService.ComputeAsync(companyId, assignmentId, vehicleId, jobId, plannedDelivery, actualDelivery, status, complianceScore) returns {etaAt, confidence, risk, reasonCodes, explanation, source, remainingMiles, effectiveSpeedMph, modelVersion}.

1. Terminal short-circuits (keep current behaviour): status=delivered+actualDelivery => actual time, high, on_time. status=cancelled => unknown. status=exception => risk=delayed, etaAt=null, reason exception_hold.

2. Gather live state (all company_id-scoped):
   pos = SELECT lat,lng,speed_mph, EXTRACT(EPOCH FROM (NOW()-GREATEST(received_at, COALESCE(device_fix_time,received_at))))::BIGINT stale_s FROM latest_vehicle_positions WHERE company_id=@c AND vehicle_id=@v.
   dest = jobs.dropoff_latitude/longitude for jobId.
   staleThreshold, circuity, defaultSpeed, minSpeedFloor, minLaneSamples from eta_config (fail-closed defaults if row absent).

3. Fail-closed gate — if pos IS NULL, or stale_s > staleThreshold, or dest coords NULL: DO NOT predict. Fall back to plannedDelivery, confidence=low, source='planned_fallback', reason in {no_telemetry, stale_telemetry, no_destination_coords}. (This is the "no config => today's numbers" analog.)

4. Remaining distance:
   a. If eta_config.use_route_api AND a Google Maps key is configured: remainingMiles = GoogleMapsService.DirectionsAsync(pos -> dest).DistanceMeters/1609.34 (cache 5 min per assignment to bound cost). Also capture road DurationSeconds as roadEtaSeconds.
   b. Else remainingMiles = Haversine(pos.lat,pos.lng,destLat,destLng) * circuity_factor (default 1.30).

5. Effective speed (mph), robust to red-light zeros:
   liveSpeed = EWMA over last 6 location_events for the vehicle in the trailing 20 min: SELECT speed_mph, event_time ... ORDER BY event_time DESC LIMIT 6; EWMA alpha=0.5, most-recent weighted highest. Ignore events with speed_mph<=1 unless ALL are <=1.
   laneSpeed = derived from eta_lane_stats.median_min_per_mile: 60.0/median_min_per_mile.
   effectiveSpeed = if liveSpeed >= minSpeedFloor: 0.6*liveSpeed + 0.4*laneSpeed (blend toward lane mean, reduces jitter); else laneSpeed (vehicle stopped/creeping — trust history). Clamp to [minSpeedFloor(8), 80].
   If eta_lane_stats has < minLaneSamples: laneSpeed = 60/eta_config.default_min_per_mile (default 1.33 => 45mph). source tag 'live' if lane samples>=min else 'live_no_lane_history'.

6. Dwell buffer: remainingStops = trip_stops for the assignment's trip with status='pending' AND stop_sequence > current; dwellMin = remainingStops * eta_lane_stats.median_stop_dwell_min (fallback 10). If no trip/stops, 0.

7. remainingMinutes = (remainingMiles / effectiveSpeed)*60 + dwellMin. If road API path (4a) succeeded, remainingMinutes = 0.5*(roadDurationMinutes) + 0.5*(computed) to fuse map ETA with live speed. etaAt = NOW() + remainingMinutes.

8. Risk: predicted vs plannedDelivery. slackMin = (plannedDelivery - etaAt) in minutes. delayed if slackMin < -15; at_risk if -15<=slackMin<30 OR openExceptions>0 OR compliance<80; on_time otherwise. openExceptions = COUNT dispatch_exceptions status='open' for assignment (existing query 16140-16143).

9. Confidence: high if stale_s<=600 AND laneSamples>=min AND openExceptions=0 AND compliance>=80; medium if stale_s<=staleThreshold AND (laneSamples>=min OR road API); low otherwise. reasonCodes accumulate: live_position, historical_lane, road_api, stale_telemetry, route_deviation, active_exceptions, sparse_lane_history, tight_window.

LANE STATS (eta_lane_stats, refreshed by background):
lane_key = concat(round(pickup_lat,1),'_',round(pickup_lng,1),'_',round(dropoff_lat,1),'_',round(dropoff_lng,1)) (~11km buckets). Per (company_id, lane_key):
  SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY actual_duration_minutes/NULLIF(actual_distance_miles,0)) median_min_per_mile,
         percentile_cont(0.9) WITHIN GROUP (...) p90_min_per_mile, COUNT(*) sample_count
  FROM trips t JOIN jobs j ON j.id=t.job_id
  WHERE t.company_id=@c AND t.status='completed' AND t.actual_distance_miles>0 AND t.actual_duration_minutes>0
  GROUP BY lane_key. Also median_stop_dwell_min from trip_stops actual_departure-actual_arrival. UPSERT ON CONFLICT(company_id, lane_key). Fleet-wide row lane_key='__fleet__' as second fallback.

ACCURACY LOOP: when status transitions to delivered, upsert eta_prediction_accuracy(company_id, dispatch_assignment_id, predicted_eta = last eta_predictions.predicted_eta, actual_delivery_at, error_minutes = |actual-predicted|, lane_key). Aggregated MAE/on-time% per lane feeds the existing "ETA Accuracy" SLA metric (referenced Batch7SchemaService.cs:432,440).

**Algorithm:**

ACCOUNTING MODEL (AP subledger -> GL), two events in the lifecycle:

(1) ACCRUE payable on approval. In GeneralLedgerService add:
  public async Task<long> PostSettlementAsync(long companyId, long statementId, CancellationToken ct=default)
  - Read scoped: SELECT status, total, currency, statement_no, approved_at, created_at FROM settlement_statements WHERE company_id=@c AND id=@id.
  - Fail-closed guards: if row is null -> throw InvalidOperationException (handler will retry/dead-letter). If status NOT IN ('approved','paid') -> return 0 and post nothing (a draft is not a recognized liability; maker-checker approval is the posting control). If total <= 0 -> return 0 (no empty/noise entry).
  - when = approved_at?.Date ?? created_at?.Date ?? DateTime.UtcNow.Date.
  - lines = [ new Line(\"5000\", total, 0m), new Line(\"2000\", 0m, total) ].
  - return await PostEntryAsync(companyId, when, \"settlement\", statementId.ToString(CultureInfo.InvariantCulture), $\"AP settlement {statement_no}\", lines, ct);
  Idempotency is free: PostEntryAsync SELECTs the existing entry for (company, 'settlement', statementId) and returns it without double-posting.

(2) SETTLE payable on payment. Add:
  public async Task<long> PostSettlementPaymentAsync(long companyId, long paymentId, CancellationToken ct=default)
  - Read scoped: SELECT amount, statement_id, paid_at, created_at FROM settlement_payments WHERE company_id=@c AND id=@id.
  - Guard: null -> throw; amount <= 0 -> return 0.
  - when = paid_at?.Date ?? DateTime.UtcNow.Date.
  - lines = [ new Line(\"2000\", amount, 0m), new Line(\"1000\", 0m, amount) ].  (Dr AP, Cr Cash)
  - return await PostEntryAsync(companyId, when, \"settlement_payment\", paymentId.ToString(...), $\"AP payment {paymentId} on statement {statementId}\", lines, ct);
  Each payment posts its own entry keyed by payment id, so partial payments accumulate and reconcile.

TRIGGER (primary, event-driven / derive-beside): SettlementService publishes durable events; new handlers post to GL.
  - In ApproveStatementAsync, after the UPDATE...status='approved' succeeds, publish: events.Publish(companyId.ToString(), \"settlement.approved\", \"settlement_statement\", statementId.ToString(), JsonSerializer.Serialize(new { statementId, total, currency }), corr, caus, idem: $\"settlement-approved-{statementId}\"). Only publish on a real draft/pending_review -> approved transition (not on the already-approved no-op at line 254), so re-approve stays a silent no-op.

  A handler throw -> outbox retries with backoff then dead-letters; the approval/payment write already committed and is never rolled back (source of truth is the subledger, GL posting is downstream + retryable) — identical guarantee to InvoiceIssuedRecognitionHandler.

**Idempotency / fail-closed:** Idempotency: GL posting is idempotent by construction - PostEntryAsync (GeneralLedgerService.cs:57-60) returns the existing journal_entries.id for a repeated (company_id, source_type, source_ref), so a redelivered outbox message, a re-approve, or a backfill after auto-post all no-op. Payment postings are keyed per paymentId, and RecordPaymentAsync already dedupes payments on idempotency_key (SettlementService.cs:286-298) so the same physical payment produces one payment row -> one GL entry. Event publish carries an idempotencyKey (settlement-approved-{id}, settlement-paid-{paymentId}) so the outbox itself dedupes enqueues. Fail-closed: PostSettlementAsync posts ONLY when status IN ('approved','paid') and total>0 - a draft or zero statement writes zero GL rows and invents no liability; no pay config -> no statement -> nothing to post (the leakage signal stays visible upstream). Every entry is balanced (Dr==Cr equal to a single amount) so PostEntryAsync's balance assertion (51-52) can never trip and the trial balance always nets to zero. Non-rollback: handlers run post-commit off the durable outbox; a GL failure throws and is retried/dead-lettered without ever reversing the approval or payment (subledger is the book of record, GL is derived) - identical to InvoiceIssuedRecognitionHandler.

**Tenant-scoping / security:** Every new read/write is scoped by explicit WHERE company_id=@c: PostSettlementAsync/PostSettlementPaymentAsync read settlement_statements/settlement_payments by (company_id,id), and PostEntryAsync already stamps company_id into journal_entries/journal_lines. This matters because the outbox dispatcher drains under the platform_admin bypass scope (OutboxDispatcherBackgroundService.cs:29-36) where RLS is not enforcing tenant isolation for the worker - so the SQL WHERE clause IS the tenant boundary, exactly as PostInvoiceAsync (90-96) already relies on. The handler derives companyId only from the event's TenantId (never from payload), so a message can only post within its own tenant. Request-path endpoints (backfill, reconciliation) use GetCompanyId(http) + RequirePermission (finance:manage / finance:view) like the existing GL endpoints. Approval remains the maker-checker control (settlement.approve, high-risk) and is the only gate that lets a liability reach the GL; nothing auto-posts from a draft. No cross-tenant account codes: chart_of_accounts is per-company (UNIQUE(company_id, account_code)).
